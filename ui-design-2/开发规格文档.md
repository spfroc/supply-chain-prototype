## 一、项目概述

### 1.1 产品定位

**政企采购行业供应链平台** — 面向政府机关、央国企及事业单位的专业采购商城，核心价值在于通过**企业协议定价**和**多平台比价**降低采购成本。

### 1.2 系统组成

| 序号 | 模块名称       | 说明                                                         |
|------|----------------|--------------------------------------------------------------|
| 1    | 文章系统(CMS)  | 资讯发布、分类管理、富文本编辑                                |
| 2    | 商城全功能模块 | 商品展示、购物车、下单、支付、订单管理                        |
| 3    | 商品专区模块   | 政采平台专区（国网/军网/齐鲁云采/国铁/青慧采），三列比价展示  |
| 4    | 批量下单模块   | 批量导入商品、批量提交订单                                    |
| 5    | 场景方案模块   | 方案组合展示、方案折扣价、多商品成套推荐                      |
| 6    | 系统配置模块   | 导航配置、管理员权限、日志、系统参数                          |

### 1.3 不纳入本期（明确排除）

| 排除项               | 原因                       |
|----------------------|----------------------------|
| 退货/退款流程        | 政企采购退货流程复杂，需单独立项 |
| 秒杀功能             | 非政企采购核心场景           |
| 拖动式方案编辑器     | 排入二期                     |
| 页面自定义设置       | 排入二期                     |

### 2.2 关键业务规则（已确认；变更须走需求评审）

| 编号 | 规则                                                         | 说明                                                                             |
|------|--------------------------------------------------------------|----------------------------------------------------------------------------------|
| R1   | 企业有有效协议 → 协议内商品显示"专属"Logo + 协议折扣价 + 可下单 | Logo 位于商品卡片右上角，红色文字，CSS transform: rotate(-45deg)                 |
| R2   | 企业无有效协议 → 可浏览不可下单，提示"暂无有效协议，请联系客服"  | 下单按钮置灰(disabled)，显示提示文案                                              |
| R3   | 协议有效期内可调整（加入新商品/移除旧商品/调整价格）            | agreement_item 表支持增删改，记录操作日志                                         |
| R4   | 平台协议价仅参考展示，不参与交易                               | platform_goods.platform_price 仅在前端"比价列"展示，下单时使用企业协议价          |
| R5   | 方案折扣 vs 企业协议折扣 → 取较低者展示                        | getSolutionPrice() 中对每个商品取 min(solutionDiscountPrice, agreementPrice)     |

### 3.2 技术栈选型（已确认）

| 层次       | 技术                     | 说明                                  |
|------------|--------------------------|---------------------------------------|
| 管理后台   | React 19.2 + TypeScript + Vite + Ant Design 6 | 管理端组件成熟，统一类型与构建工具 |
| Web 客户端 | React 19.2 + TypeScript + Vite | SPA 架构，与管理端共享类型、请求层和设计 Token |
| H5 客户端  | React 19.2 + TypeScript + Vite + Ant Design Mobile 5 | 响应式 H5，与 Web 共享业务模型 |
| 后端 API   | Java 21 LTS + Spring Boot 4.1 + Maven | 模块化单体优先，后续按业务压力拆服务 |
| 数据库     | MySQL 8.4 LTS             | InnoDB、utf8mb4，选择长期支持版本        |
| 缓存       | Redis 8.x                 | 价格缓存、Token 会话、分布式锁           |
| 文件存储   | 阿里云 OSS                | 商品图、方案配图、富文本图片和导入文件    |
| 反向代理   | Nginx                    | 静态资源 + API 代理                    |

**工程约束**：采用前后端 Monorepo；前端包管理器统一 `pnpm`，接口契约以 OpenAPI 3.1 为唯一事实来源。首期采用模块化单体，不引入微服务、消息队列和 Kubernetes；异步任务先用数据库任务表 + 调度器实现。

### 3.3 缓存策略

|------------------|--------|-----------------------------------------|

## 四、数据库设计

### 4.1 命名规范

- 表名：小写 + 下划线，如 `agreement_item`
- 字段名：小写 + 下划线，如 `created_at`
- 主键：统一使用 `id` (BIGINT UNSIGNED AUTO_INCREMENT)
- 时间字段：统一 `created_at` / `updated_at` (DATETIME)
- 删除标记：统一 `deleted_at` (DATETIME NULL) — 软删除

## 五、状态机定义

### 5.1 企业状态 (enterprise.status)

```
                    ┌─────────┐
        注册/提交    │  待审核  │
      ┌───────────→ │  (0)    │
      │             └────┬────┘
      │                  │
      │        ┌─────────┼─────────┐
      │        │ 审核通过  │ 审核驳回  │
      │        ▼         │ (回到待审核, │
      │   ┌─────────┐   │  记录驳回原因)│
      │   │  已审核  │   │              │
      │   │  (1)    │←──┘              │
      │   └────┬────┘                  │
      │        │ 管理员禁用              │
      │        ▼                       │
      │   ┌─────────┐                  │
      │   │  已禁用  │                  │
      │   │  (2)    │                  │
      │   └────┬────┘                  │
      │        │ 管理员重新启用           │
      │        └────────┐              │
      │                 ▼              │
      │            ┌─────────┐         │
      └────────────│  已审核  │←────────┘
                   │  (1)    │
                   └─────────┘

变更规则:
- 待审核(0) → 已审核(1): 管理员操作,记录 audit_remark + audited_at
- 待审核(0) → 待审核(0): 驳回,记录 audit_remark
- 已审核(1) → 已禁用(2): 管理员禁用
- 已禁用(2) → 已审核(1): 管理员重新启用
- 已审核(1) → 待审核(0): 不允许
- 已禁用(2) → 待审核(0): 不允许
```

### 5.2 商品(SPU)状态 (spu.status)

```
                   ┌─────────┐
       创建        │  待审核  │
     ┌───────────→ │  (0)    │
     │             └────┬────┘
     │                  │
     │        ┌─────────┼─────────┐
     │        │ 审核通过  │ 审核驳回  │
     │        ▼         │ (回到待审核)│
     │   ┌─────────┐   │              │
     │   │  已审核  │←──┘              │
     │   │  (1)    │                  │
     │   └────┬────┘                  │
     │        │                       │
     │   ┌────┼────┐                  │
     │   │         │                  │
     │   ▼         ▼                  │
     │ ┌─────┐  ┌─────┐              │
     │ │无效 │  │已下市│              │
     │ │(2) │  │(3)  │              │
     │ └──┬──┘  └──┬──┘              │
     │    │        │                  │
     │    │ 重新审核│ 重新审核          │
     │    └────┬───┘                  │
     │         ▼                      │
     │    ┌─────────┐                 │
     └────│  已审核  │←────────────────┘
          │  (1)    │
          └─────────┘

变更规则:
- 待审核(0) → 已审核(1): 管理员操作
- 已审核(1) → 无效(2): 商品信息错误/违规
- 已审核(1) → 已下市(3): 商品停售
- 无效(2) → 已审核(1): 重新审核通过
- 已下市(3) → 已审核(1): 重新上架
- 已审核(1) → 待审核(0): 不允许
```

### 5.3 协议状态 (agreement.status)

```
                    ┌─────────┐
        创建        │  待生效  │
     ┌───────────→  │  (0)    │
     │              └────┬────┘
     │                   │
     │          ┌────────┼────────┐
     │          │ start_date      │ end_date
     │          │ 到达(定时任务)    │ 到达(定时任务)
     │          ▼                 ▼
     │     ┌─────────┐      ┌─────────┐
     │     │  生效中  │      │  已过期  │
     │     │  (1)    │─────→│  (2)    │
     │     └─────────┘      └─────────┘
     │
     └── 协议有效期内可调整商品明细(agreement_item增删改)

定时任务逻辑(每日0点执行):
  UPDATE agreement SET status=1 WHERE status=0 AND start_date <= CURDATE()
  UPDATE agreement SET status=2 WHERE status=1 AND end_date < CURDATE()

变更规则:
- 待生效(0) → 生效中(1): 系统自动(定时任务)
- 生效中(1) → 已过期(2): 系统自动(定时任务)
- 不允许手动从 生效中(1) 变更为其他状态
- 不允许从 已过期(2) 变更为其他状态
```

### 5.4 订单状态 (order_main.order_status)

```
创建订单 ──→ 待付款(1) ──后台确认银行到账──→ 待发货(2)
                 │                              │
                 │ 用户/管理员取消              │ 管理员发货
                 ▼                              ▼
             已取消(5)                     待收货(3)
                                                │ 用户确认或超时确认
                                                ▼
                                           已完成(4)

变更规则:
- 创建成功直接进入待付款(1)，企业订单无需审核、无分级审批
- 待付款(1) → 待发货(2): 财务管理员核对银行到账后手动确认
- 待付款(1) → 已取消(5): 用户取消、管理员取消或超过付款期限自动取消（默认7天，可配置）
- 待发货(2) → 待收货(3): 管理员发货
- 待收货(3) → 已完成(4): 用户确认收货 或 超时自动确认(默认15天)
- 已完成(4) 不可变更
- 已取消(5) 不可变更
```

### 5.5 支付状态 (order_main.pay_status)

```
待确认(0) ──财务管理员确认到账──→ 已确认(1)
   │                                │
   │ 订单取消                       ├──后台登记部分退款──→ 部分退款(2)
   │（状态保持0）                    └──后台登记全额退款──→ 已退款(3)

变更规则:
- 不对接线上支付渠道，无支付回调；客户按订单页展示的公司银行账户线下转账
- 待确认(0) → 已确认(1): 财务管理员录入银行流水参考号、到账金额、到账时间并确认
- 实际到账金额必须等于订单应付金额；不一致时不得确认，转人工处理
- 已确认(1) → 部分退款(2)/已退款(3): 仅管理后台保留人工登记能力，不执行自动出款
- 部分退款(2) → 已退款(3): 管理员登记剩余退款
- 已退款(3) 不可变更
- 所有支付/退款状态修改必须记录操作日志，禁止客户端修改
```

### 5.6 发票状态 (invoice.status)

```
已申请(1) ──→ 第三方处理中(2) ──→ 已开具(3)
                           └──→ 开具失败(4)
已开具(3) ──第三方完成红冲──→ 已红冲(5)

变更规则:
- 未申请时不创建记录；已付款订单可提交申请并创建状态1记录
- 平台不负责开具发票，管理员将申请提交给第三方后标记为处理中(2)
- 第三方开具完成后，管理员在平台录入发票代码、号码、电子票文件和开票时间，标记为已开具(3)
- 第三方开具失败或完成红冲后，由管理员同步登记结果并填写原因
- 平台本期不对接第三方发票 API，不自动开票
```

## 七、价格计算引擎

### 7.1 核心价格计算伪代码

```
/**
 * 获取用户对指定 SKU 的最终价格
 * 调用时机: SPU详情、SKU列表、购物车、方案详情、提交订单
 * 调用位置: 服务端(价格计算统一在服务端完成)
 */
function getUserPrice(user, sku):
    // 1. 未登录 → 返回通用会员价
    if user == null:
        return {
            price: sku.memberPrice,
            priceType: 'member',
            canOrder: false,
            label: '会员价'
        }

    // 2. 获取用户所属企业
    enterprise = user.enterprise
    if enterprise == null:
        return {
            price: sku.memberPrice,
            priceType: 'member',
            canOrder: false,
            label: '会员价'
        }

    // 3. 查找企业有效期内协议（缓存30分钟）
    agreement = findValidAgreement(enterprise.id)
    // findValidAgreement: SELECT * FROM agreement WHERE enterprise_id=? AND status=1 AND start_date<=NOW() AND end_date>=NOW() LIMIT 1

    // 4. 无有效协议 → 可浏览不可下单
    if agreement == null:
        return {
            price: sku.memberPrice,
            priceType: 'member',
            canOrder: false,
            label: '会员价',
            tip: '暂无有效协议，请联系客服'
        }

    // 5. 查找协议内商品
    item = findAgreementItem(agreement.id, sku.id)
    // findAgreementItem: SELECT * FROM agreement_item WHERE agreement_id=? AND sku_id=? LIMIT 1

    // 6. 商品不在协议内 → 通用会员价，不可下单
    if item == null:
        return {
            price: sku.memberPrice,
            priceType: 'member',
            canOrder: false,
            label: '会员价',
            tip: '该商品不在当前协议范围内'
        }

    // 7. 有协议价 → 返回协议折扣价
    return {
        price: item.discountPrice,
        priceType: 'agreement',
        canOrder: true,
        label: '协议折扣价',
        marketPrice: sku.marketPrice,
        memberPrice: sku.memberPrice,
        discountRate: item.discountRate,
        isExclusive: true     // 前端据此显示"专属"Logo
    }


/**
 * 获取方案价格(每个商品取较低者)
 * 调用时机: 方案详情页
 */
function getSolutionPrice(user, solution):
    items = []

    for each item in solution.items:
        sku = item.sku

        // 获取协议价
        agreementResult = getUserPrice(user, sku)

        // 方案折扣价 vs 协议折扣价 → 取较低者
        if agreementResult.canOrder and agreementResult.price < item.solutionDiscountPrice:
            finalPrice = agreementResult.price
            isAgreementBetter = true
        else:
            finalPrice = item.solutionDiscountPrice
            isAgreementBetter = false

        items.push({
            skuId: sku.id,
            skuName: sku.name,
            skuImage: sku.image,
            specDesc: sku.specDesc,
            quantity: item.quantity,
            roleDescription: item.roleDescription,
            solutionDiscountPrice: item.solutionDiscountPrice,
            agreementPrice: agreementResult.price,           // null if no agreement
            finalPrice: finalPrice,
            isAgreementBetter: isAgreementBetter,
            installPrice: item.installPrice,
            afterSalesService: item.afterSalesService,
            technicalSupport: item.technicalSupport,
            subtotal: finalPrice * item.quantity + item.installPrice
        })

    totalPrice = sum of all items.subtotal

    return {
        solution: solution,
        items: items,
        totalPrice: totalPrice
    }


/**
 * 计算订单总金额
 * 调用时机: 提交订单、购物车结算预览
 */
function calculateOrderTotal(user, orderItems):
    totalAmount = 0

    for each orderItem in orderItems:
        priceResult = getUserPrice(user, orderItem.sku)

        if !priceResult.canOrder:
            throw BusinessError('商品 [' + orderItem.sku.name + '] 不在协议范围内，无法下单')

        subtotal = priceResult.price * orderItem.quantity
        orderItem.unitPrice = priceResult.price
        orderItem.subtotal = subtotal
        totalAmount += subtotal

    // 应用优惠券(如有)
    if orderItem.couponId != null:
        totalAmount = applyCoupon(totalAmount, orderItem.couponId)

    return totalAmount
```

## 八、登录鉴权流程

### 8.1 JWT Token 方案

```
Token 结构:
Header:  {"alg": "HS256", "typ": "JWT"}
Payload: {
    "sub": 1,                    // user.id
    "phone": "138****8888",
    "enterprise_id": 1,
    "role": "main_account",
    "iat": 1721558400,          // 签发时间
    "exp": 1721565600           // 过期时间(2小时)
}
Signature: HMAC-SHA256(base64UrlEncode(header) + "." + base64UrlEncode(payload), secret)

Token 存储:
- 服务端: Redis key `token:{token}` → userInfo (TTL=2h, 与JWT过期一致)
- 客户端：Web 使用 Secure、HttpOnly、SameSite Cookie；H5 WebView 沿用 Cookie 会话（不得写入 localStorage）
```

### 8.2 登录流程

```
客户端                          服务端                         数据库/缓存
  │                              │                              │
  │  POST /api/v1/auth/login     │                              │
  │  {phone, password}           │                              │
  │ ──────────────────────────→ │                              │
  │                              │  查询user                     │
  │                              │ ──────────────────────────→ │
  │                              │ ←────────────────────────── │
  │                              │                              │
  │                              │  验证password_hash (bcrypt)  │
  │                              │  查询enterprise               │
  │                              │ ──────────────────────────→ │
  │                              │ ←────────────────────────── │
  │                              │                              │
  │                              │  查询有效协议                  │
  │                              │ ──────────────────────────→ │
  │                              │ ←────────────────────────── │
  │                              │                              │
  │                              │  生成JWT Token                │
  │                              │  存入Redis (token→userInfo)   │
  │                              │ ──────────────────────────→ │
  │                              │                              │
  │  返回 {token, user,          │                              │
  │        enterprise,           │                              │

  │ ←────────────────────────── │                              │
```

### 8.3 请求鉴权中间件

```
function authMiddleware(request, response, next):
    token = extractToken(request.headers['Authorization'])

    if token == null:
        return response 401 {code: 3001, message: '未登录'}

    // 从Redis获取用户信息
    userInfo = redis.get('token:' + token)

    if userInfo == null:
        return response 401 {code: 3002, message: 'Token已过期'}

    // 注入到请求上下文
    request.currentUser = userInfo

    next()
```

### 8.4 价格计算安全原则

- **所有价格计算在服务端完成**，前端仅负责展示
- Token 解析后获取 `enterprise_id`，服务端据此查询协议和价格
- 前端不存储任何价格计算逻辑
- 提交订单时服务端重新计算价格并校验，防止前端篡改
- 提交订单时校验企业协议状态（防止协议过期后下单）

## 十四、详细业务需求补充

> 本章补齐原型中已出现、但原规格未形成闭环的规则。若与前文冲突，以本章为准。

### 14.1 用户、企业与登录

1. 企业注册时创建企业主账号，手机号验证码通过后才能提交；统一社会信用代码和手机号分别唯一。
2. 注册申请状态为：`draft`（草稿）→ `pending`（待审核）→ `approved`（通过）/ `rejected`（驳回）。驳回后允许修改资料再次提交，并保留每次审核记录。
3. 企业审核通过不等于具备下单权。下单权还必须同时满足：企业启用、用户启用、存在有效协议、目标 SKU 位于该协议中、SKU 上架且库存满足要求。
4. 主账号可新增、停用企业子账号；子账号仅能访问本企业数据。管理员账号与客户端用户账号使用不同登录入口、Token 受众和权限体系。
5. 登录连续失败 5 次后锁定账号 15 分钟；密码不少于 8 位且必须同时包含字母和数字。找回密码通过短信验证码完成。
6. Access Token 有效期 2 小时，Refresh Token 有效期 7 天；退出登录、账号停用或改密后，服务端立即撤销相关 Token。

### 14.2 协议生效与价格规则

1. 有效协议判定统一为：`status=1 AND start_date <= 当前业务日期 AND end_date >= 当前业务日期 AND deleted_at IS NULL`。
2. 同一企业同一时刻只允许一个生效中的协议；创建或调整协议时必须检测日期重叠并拒绝冲突。生效协议可随时添加/删除商品和调整价格，所有变更实时生效、主动失效缓存并写入操作日志。
3. `discount_rate` 采用百分数语义：`85.00` 表示按会员价的 85% 计价，即俗称“8.5 折”；前文“85折”表述统一更正为“8.5折”。
4. 协议商品价格来源只能二选一：固定 `discount_price`，或由 `member_price × discount_rate / 100` 计算。v1.1 默认以固定 `discount_price` 为成交依据，`discount_rate` 仅用于展示和校验。
5. 方案下单时逐个 SKU 取 `min(solution_discount_price, agreement_price)`；安装费不参与取低，按方案项独立累加。最终金额公式：`Σ(成交单价 × 数量 + 安装费 × 数量) - 优惠金额`。
6. 所有金额字段单位统一为人民币元，数据库使用 `DECIMAL`，计算过程保留 4 位小数，最终按四舍五入保留 2 位；禁止使用浮点数。
7. 协议或会员价变更后立即删除对应价格缓存；商品详情、购物车、确认订单、提交订单均返回 `price_version`。提交时版本不一致返回 `409 PRICE_CHANGED` 并要求用户确认新价格。
8. 商品卡片的“专属”标识只表示当前企业对该 SKU 存在有效协议价，不表示商品或企业整体身份。

### 14.3 商品与库存

1. SPU 至少包含一个有效 SKU 才能上架；SKU 编码全局唯一。分类、品牌禁用后不得新建关联商品，已关联商品保留历史展示但不可新上架。
2. 下单数量必须为 `min_order_qty` 的整数倍，且大于 0、不超过可售库存。购物车展示库存变化，但仅在提交订单时做最终校验。
3. 商品全部使用平台自营单仓库存，不设置供应商和仓库维度。提交订单成功后在同一事务中占用库存；订单取消或超时关闭后释放；财务确认银行到账后将占用量转为已售量。
4. 商品下架、协议移除、库存不足时，购物车条目标记失效且不参与全选和结算，页面给出具体原因。
5. 商品详情和订单快照必须记录 SPU/SKU 名称、编码、规格、图片、成交价类型、协议 ID、方案 ID、税率（如启用）与价格版本，保证历史订单可追溯。

### 14.4 购物车

1. 购物车按用户隔离；同一 SKU 且同一购买上下文（普通购买/方案购买）合并数量，不同方案来源不合并。
2. 加入购物车、修改数量、勾选、删除均需校验 SKU 状态。购物车最多 200 个有效条目，单次结算最多 100 个条目。
3. 登录后服务端保存购物车；本期不合并匿名购物车。批量删除和失效商品清理必须幂等。
4. 进入确认订单页时生成有效期 30 分钟的结算预览 `checkout_token`，锁定所选条目、地址分配和价格版本，但不锁定最终成交价。

### 14.5 订单、审核、支付与履约

1. 创建订单接口必须接收客户端幂等键 `Idempotency-Key`；同一企业、用户和幂等键只生成一个主订单。
2. 一个主订单可按收货地址拆分多个子订单。同一 SKU 可拆分数量到多个地址；请求以 `allocations` 明确每个地址的分配数量，且各地址数量之和必须等于购买总量。
3. 企业订单无需审核且无分级审批，创建成功直接进入 `待付款`。
4. 用户取消或超过付款期限后进入 `已取消`并释放库存。线下银行转账默认付款期限为 7 天，可由系统参数配置。
5. 订单页展示平台收款账户、应付金额、订单号和转账备注要求。财务管理员核对银行流水后，在后台填写银行流水参考号、到账金额、到账时间和备注，手动确认到账。
6. 确认到账接口必须幂等，且到账金额须与应付金额一致；成功后同步将支付状态改为已确认、订单改为待发货、占用库存转为已售。
7. 管理后台保留部分退款/已退款状态和人工登记入口，不对接自动出款；登记必须填写退款金额、银行流水参考号、时间与原因并写操作日志。
8. 子订单只支持一个物流单，不支持多包裹或分批发货；物流字段继续保留在 `order_sub`。
9. 管理员只能对 `待发货`子订单执行发货；全部子订单签收后主订单自动完成，也允许管理员在异常情况下人工完成并记录原因。
10. 订单详情按数据归属校验：客户端用户只能访问本企业订单；普通子账号默认只能访问自己创建的订单，主账号可访问本企业全部订单。

### 14.6 多地址确认订单

1. 默认使用用户默认地址；没有默认地址时要求用户选择。每个用户最多维护 20 个未删除地址且只能有一个默认地址。
2. 地址删除不影响历史订单。下单时将联系人、电话和完整地址写入子订单地址快照，履约读取快照而非实时地址表。
3. 同一 SKU 拆至多个地址时，每个地址生成独立订单商品行；后端校验分配总量、最小起订量与总库存，禁止负数、零数量和超额分配。
4. 更改地址或分配数量后重新计算配送可达性、运费及结算预览。当前版本默认运费为 0；未来启用运费时不得改变历史订单金额。

### 14.7 优惠券

1. 优惠券领取、核销在事务内完成；`used_count` 不得超过 `total_count`，同一张用户券只能核销一次。
2. 优惠券不得与协议价或方案价叠加。结算商品只要使用了协议价或方案价，整个订单不可选择优惠券，并明确提示原因。
3. 由于本期只有协议内商品可以下单，优惠券在本期客户端结算流程中实际不可使用；保留优惠券管理、领取记录和后续普通价下单扩展能力，但隐藏领取及使用入口。
4. 未来开放普通价下单后，单个主订单最多使用一张优惠券，优惠金额不得超过订单商品金额，取消未付款订单时返还优惠券。

### 14.8 发票

1. 仅已支付订单可申请发票，单个订单只允许一个有效发票申请；开票金额不得超过订单实付金额。
2. 发票状态调整为：`1=已申请、2=开具中、3=已开具、4=开具失败、5=已红冲`；未申请不创建记录。
3. 增值税专用发票必须填写开户行、银行账号、注册地址、注册电话和接收邮箱；普通发票至少填写抬头、税号和接收邮箱。
4. 发票由第三方线下开具，平台仅记录申请与结果，不对接开票 API。开具后由管理员录入发票代码、号码、电子票文件地址及开票时间；客户端只允许下载本企业发票。

### 14.9 批量下单

1. 支持 `.xlsx` 模板，必填列为 SKU 编码、数量、收货人、手机号、省、市、区、详细地址；单文件最大 5 MB、最多 1,000 行。
2. 上传后先异步校验，不直接创建订单。校验结果逐行返回：行号、字段、错误码、错误说明；整批无错误或用户剔除错误行后才可提交。
3. 同一文件摘要在 10 分钟内重复上传视为同一任务，避免重复下单。批量任务应记录上传人、总行数、成功数、失败数和结果文件。
4. 批量订单仍遵循协议、价格、库存、线下付款、幂等和多地址规则；同一地址的行合并到同一子订单。

### 14.10 后台审计与敏感操作

1. 企业审核、协议创建/调价/移除商品、商品上下架、库存调整、银行到账确认、退款登记、发货/人工完成、发票结果登记、角色权限变化均必须写操作日志。
2. 日志至少记录操作者、模块、动作、目标类型与 ID、变更前后摘要、IP、请求 ID、结果和时间；敏感字段只记录脱敏值。
3. 价格批量修改应先展示影响 SKU 数、变更前后差异，二次确认后提交；单次最多 5,000 条，失败时整体回滚。

## 十六、非功能需求

### 16.1 性能与容量

| 指标 | 验收目标 |
|---|---|
| 普通查询 API | P95 ≤ 500 ms，P99 ≤ 1 s（不含第三方接口） |
| 提交订单 API | P95 ≤ 2 s |
| 商品列表 | 默认 20 条/页，最大 100 条/页 |
| 并发基线 | 300 并发用户、50 TPS 下单峰值；上线前按真实容量压测修订 |
| 可用性 | 月度服务可用性 ≥ 99.9% |
| 数据备份 | 数据库每日全量、每小时增量；RPO ≤ 1 小时，RTO ≤ 4 小时 |

### 16.2 安全与合规

1. 全站 HTTPS；密码使用 BCrypt（cost ≥ 12）或 Argon2id；Token、API 密钥、银行账号等敏感数据不得明文写日志。
2. 不将 Access/Refresh Token 存入 `localStorage`。Web 端推荐 `Secure + HttpOnly + SameSite` Cookie；若使用 Authorization Header，Access Token 仅存内存，Refresh Token 存 HttpOnly Cookie。
3. 所有写接口启用鉴权、权限校验、数据归属校验、参数白名单和请求频率限制；Cookie 方案同时启用 CSRF 防护。
4. 富文本入库及展示前做 HTML 白名单清洗；上传文件校验 MIME、扩展名、大小并使用随机文件名，禁止脚本文件直接公开访问。
5. 手机号、身份证件、银行账号在界面和日志中脱敏；生产数据原则上不得进入测试环境。
6. 管理员高风险操作建议启用 MFA；操作日志至少保留 2 年，订单与财务数据按适用法规及合同约定保留。

### 16.3 可观测性与运维

1. 每个请求生成 `request_id` 并贯穿网关、应用日志、异步任务和第三方回调。
2. 监控 API 错误率、P95 延迟、数据库连接池、Redis 命中率、库存占用异常、银行到账确认异常、物流回调失败和批量任务积压。
3. 关键告警分级：支付/订单无法创建为 P1，价格缓存或批量任务异常为 P2，内容管理异常为 P3。
4. 发布需具备数据库迁移脚本、回滚说明和灰度/健康检查；迁移脚本必须可重复检测且不可静默丢数据。

### 16.4 兼容性与可访问性

1. Web 支持当前及前一个大版本的 Chrome、Edge；H5 支持 iOS 15+ Safari、Android 10+ 的主流 WebView。
2. Web 端基准宽度 1280 px，H5 适配 320–430 px；金额、按钮和表格在系统字体放大 200% 时仍可操作。
3. 表单控件需有可读标签、错误提示与键盘焦点；关键状态不能只依赖颜色区分。

## 十七、验收标准

### 17.1 核心业务验收场景

| 编号 | 前置条件 | 操作 | 预期结果 |
|---|---|---|---|
| AC-01 | 企业审核通过但无有效协议 | 浏览商品并尝试结算 | 可浏览会员价；结算禁用并提示联系客服 |
| AC-02 | 企业存在有效协议，SKU 在协议内 | 查看商品详情 | 展示专属标识与协议价，可加入购物车 |
| AC-03 | 协议价低于方案价 | 从方案下单 | 逐项采用协议价，并标注“协议价更优” |
| AC-04 | 方案价低于协议价 | 从方案下单 | 逐项采用方案价；安装费独立累加 |
| AC-05 | 购物车页面打开后协议价变化 | 提交订单 | 返回 409 和新旧价格，不生成重复订单 |
| AC-06 | 库存仅剩 1 件，两个请求同时购买 | 并发提交 | 仅一个成功占库，库存不为负数 |
| AC-07 | 使用相同幂等键重复提交 | 连续调用创建订单 | 返回同一订单，不重复占库或核销优惠券 |
| AC-08 | 同一 SKU 购买 10 件并按 2/3/5 分到三个地址 | 提交订单 | 生成一个主订单、三个子订单及三条数量正确的商品行，总量仍为10 |
| AC-09 | 财务管理员重复确认同一银行流水 | 两次提交确认到账 | 第二次幂等返回原结果，状态和库存只推进一次 |
| AC-10 | 未支付订单超时 | 定时任务执行 | 订单取消、库存释放、优惠券返还 |
| AC-11 | 子账号访问其他企业订单 ID | 请求订单详情 | 返回 403 或 404，不泄露订单内容 |
| AC-12 | 上传含错误行的批量模板 | 等待校验完成 | 返回逐行错误；确认前不创建任何订单 |
| AC-13 | 申请专票但缺银行账号 | 提交发票申请 | 返回字段级校验错误，不创建申请 |
| AC-14 | 管理员修改协议商品价格 | 保存成功 | 新价生效、缓存失效、操作日志含前后差异 |
| AC-15 | 协议价或方案价订单尝试使用优惠券 | 生成结算预览 | 优惠券不可选，返回明确的不叠加原因 |

### 17.2 交付完成定义（Definition of Done）

1. 功能通过产品验收、接口自动化测试与关键并发测试；核心模块单元测试覆盖率不低于 80%。
2. OpenAPI 文档与实现一致，包含示例、错误码、鉴权要求、幂等约定和分页结构。
3. 数据库迁移、初始化数据、部署配置模板、监控面板和运维手册齐备。
4. 不存在阻断级或严重级安全问题；高风险接口完成越权、重放、注入和并发校验。
5. Web/H5/管理端的空态、加载态、错误态、无权限态、网络重试和表单校验均有实现。

## 十八、确认结论与外部服务选型

### 18.1 业务确认结论

Q1–Q12 已于 2026-07-21 全部确认，结论已同步写入技术栈、状态机、API、数据模型和详细业务规则。本章仅保留决策追溯：

| 编号 | 确认结论 |
|---|---|
| Q1 | Java 21 LTS + Spring Boot 4.1；三端统一 React 19.2 + TypeScript + Vite |
| Q2 | 线下银行转账，财务管理员在后台核对流水并手工确认到账 |
| Q3 | 企业订单无需审核，没有分级审批 |
| Q4 | 同一企业同一时刻仅一个协议生效；生效协议可随时增删商品和调价 |
| Q5 | 全部自营库存，不需要仓库维度 |
| Q6 | 优惠券不与协议价或方案价叠加 |
| Q7 | 渠道管理属于二期供应商/第三方货源管理，本期不实现 |
| Q8 | 同一 SKU 允许按数量拆分到多个地址 |
| Q9 | 一个子订单一个物流单，不支持多包裹 |
| Q10 | 发票由第三方开具，平台只记录申请和结果 |
| Q11 | 使用自有短信接口、阿里云 OSS；地图和物流采用下节推荐方案 |
| Q12 | 管理后台保留部分退款/已退款状态及人工登记入口，不对接自动退款 |

### 18.2 地图服务选型：高德地图

推荐使用**高德地图开放平台**：本期主要用于省市区地址标准化、地址转经纬度和配送地址校验，服务端接入 Web 服务 API 即可；若后续需要在 Web/H5 上拖动选点，再申请 Web 端（JS API）Key。

申请事项：

1. 注册并完成开发者认证，在控制台创建“政企采购供应链平台”应用。
2. 首期创建服务平台为“Web服务”的 Key，开通地理/逆地理编码能力；Key 只保存在后端配置中心。
3. 配置服务端 IP 白名单与调用配额告警，生产、测试环境使用不同 Key。
4. 若后续启用前端 JS 地图，单独创建 JS API Key 并配置域名白名单和安全密钥，不复用服务端 Key。

官方入口：[高德 Web 服务 API Key 申请](https://lbs.amap.com/api/webservice/guide/create-project/get-key)、[地理/逆地理编码文档](https://lbs.amap.com/api/webservice/guide/api/georegeo/)。

### 18.3 物流查询选型：快递100企业 API

推荐使用**快递100企业 API**，避免分别对接顺丰、京东物流、德邦、中通等承运商。本期建议申请以下两项：

1. “快递信息实时查询 API”：供用户在订单详情页主动查询，单号查询结果缓存 30 分钟，禁止高频轮询。
2. “快递信息订阅推送 API”：供系统自动接收轨迹和签收状态；回调必须验签、幂等，并保存最近一次原始响应摘要。

申请企业账号后获取 API Key/Secret，生产与测试凭证隔离。物流能力通过 `LogisticsProvider` 适配器封装，未来可替换快递鸟或承运商直连而不影响订单领域逻辑。

官方入口：[快递100 API 文档中心](https://api.kuaidi100.com/document/)、[实时查询接口](https://s.apifox.cn/apidoc/docs-site/514987/api-8562559)。

### 18.4 外部接口配置要求

1. 自有短信接口应提供发送、状态查询/回执、签名和模板编码；验证码有效期 5 分钟，同一手机号 60 秒内最多发送一次、每日最多 10 次。
2. 阿里云 OSS 使用私有 Bucket；上传与下载通过短时效签名 URL，服务端账号遵循最小权限，禁止将 AccessKey 写入前端。
3. 所有外部服务通过适配器接口接入，开发/测试环境提供 Mock 实现；超时、重试、熔断、限流和费用告警统一配置。
